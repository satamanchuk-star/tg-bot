# AlexBot

Telegram-бот для управления форумом жилого комплекса. Построен на Python 3.12 и aiogram 3.4.1.

## Возможности

- **Модерация**: страйки, мут, бан пользователей
- **Игры**: блэкджек «21» со ставками (5/10/25/50 монет) в теме «Игры», окно 22:00–00:00 МСК; стартовый баланс 200, /бонус +10 в сутки, спасение банкрота, история партий
- **Статистика**: отслеживание активности по топикам форума
- **Фильтрация**: автоматическая проверка на мат и флуд
- **RAG-база знаний**: расширенные категории (включая детскую площадку, коммунальные сервисы, доступ и платежи) и приоритет админских записей в контексте
- **Тональный анализ грубостей**: распознавание маскированного мата и мягкие предупреждения (`warning`) для сообщений средней агрессии
- **Память диалога**: краткий summary последних реплик перед ответом ассистента
- **Инфраструктура ЖК**: импорт объектов из Google Sheets в БД и использование этих данных в ответах ассистента

## Архитектура

```
app/
├── main.py              # Точка входа, настройка бота и планировщика
├── config.py            # Конфигурация через переменные окружения
├── db.py                # Async SQLAlchemy + SQLite
├── models.py            # ORM-модели
├── handlers/            # Роутеры aiogram
│   ├── admin.py         # /mute, /ban, /strike, /addcoins
│   ├── blackjack.py     # /21 со ставками, /бонус, /score, /21top, /подарить
│   ├── moderation.py    # Фильтрация контента
│   └── ...
├── services/            # Бизнес-логика
│   ├── blackjack.py     # Логика «21», выплаты, история партий
│   ├── coins.py         # Экономика монет (баланс, бонус, переводы)
│   ├── strikes.py       # Управление страйками
│   └── ...
└── utils/               # Вспомогательные функции
```

**Ключевые принципы:**
- Полностью асинхронный код (async/await)
- Разделение на handlers (обработка команд) и services (бизнес-логика)
- Планировщик APScheduler для периодических задач (лидерборд, heartbeat)

## Конфигурация

Переменные окружения в `.env` (см. `.env.example`):

| Переменная | Описание |
|------------|----------|
| `BOT_TOKEN` | Токен бота от BotFather |
| `FORUM_CHAT_ID` | ID форума (supergroup) |
| `ADMIN_LOG_CHAT_ID` | Куда слать логи админу |
| `DATABASE_URL` | Строка подключения к БД (по умолчанию SQLite) |
| `TIMEZONE` | Часовой пояс (по умолчанию Europe/Moscow) |
| `BUILD_VERSION` | Версия сборки для логов (по умолчанию dev) |
| `AI_*` | Настройки AI endpoint, таймаутов, ретраев и лимитов |
| `AI_FEATURE_*` | Точечное отключение AI-функций (модерация/ассистент/сводка) |
| `TOPIC_*` | ID топиков форума (см. `.env.example`) |
| `GOOGLE_SHEETS_SPREADSHEET_ID` | ID таблицы Google Sheets для импорта |
| `GOOGLE_SHEETS_WORKSHEET_NAME` | Имя листа для импорта (по умолчанию `Objects`) |
| `GOOGLE_SERVICE_ACCOUNT_FILE` | Путь к JSON-файлу service account |

## Импорт инфраструктуры из Google Sheets

1. Установить зависимости:

```bash
pip install -r requirements.txt
```

2. Создать Service Account в Google Cloud Console и скачать JSON-ключ.
3. Выдать доступ service account к таблице Google Sheets (как обычному пользователю по email аккаунта).
4. Положить JSON-ключ в безопасное место на сервере (например `/opt/alexbot/secrets/google-service-account.json`).
5. Заполнить env-переменные:
   - `GOOGLE_SHEETS_SPREADSHEET_ID=1OsPh54Bn5fdkfsEJyKcbYZTcHnue3EWQ`
   - `GOOGLE_SHEETS_WORKSHEET_NAME=Objects`
   - `GOOGLE_SERVICE_ACCOUNT_FILE=/opt/alexbot/secrets/google-service-account.json`
6. Запустить импорт:

```bash
python scripts/import_places_from_google_sheets.py
```

Для безопасной проверки без записи в БД:

```bash
python scripts/import_places_from_google_sheets.py --dry-run
```

Проверка загрузки данных:
- задать ассистенту вопрос про объект/категорию/адрес инфраструктуры (например, про МФЦ или ближайшие магазины);
- убедиться, что в ответе используются данные из локальной БД.

## AI: текущий статус

Бот поддерживает два режима AI:
- **Remote AI**: если задан `ANTHROPIC_API_KEY` (алиасы: `AI_KEY`/`AI_API_KEY`), бот ходит напрямую в **Anthropic Claude** (Messages API через официальный `anthropic` SDK). По умолчанию всё на дешёвом `claude-haiku-4-5`, премиум-путь — `claude-sonnet-4-6`.
- **Stub fallback**: если ключ не задан или API недоступен, бот автоматически работает в локальном режиме без падений.

В remote-режиме включены:
- модерация (структурированное решение: тип/severity/action);
- ассистент (`/ai`, упоминания, reply) с ограничением ответа до 800 символов;

- AI-резюме в ежедневной сводке.

Дополнительно учитываются суточные лимиты запросов/токенов (`AI_DAILY_REQUEST_LIMIT`, `AI_DAILY_TOKEN_LIMIT`).

## Что важно знать администраторам чата (акцент на AI)

Сейчас бот работает как «гибрид»: AI-функции включаются при рабочем ключе и runtime-флаге, но при любой проблеме бот не падает и уходит в локальный fallback.

### Что делает AI прямо сейчас

- **AI-модерация сообщений**
  - Анализирует текст с учётом контекста и возвращает структурированное решение: тип нарушения, severity, confidence и рекомендованное действие.
  - Логика настроена консервативно: при сомнениях выбирается «не наказывать».
- **AI-ассистент для жителей**
  - Отвечает по команде `/ai`, в reply и при упоминании бота.
  - Ответы короткие (до 800 символов), с ограничениями по небезопасным темам.
  - При наличии базы знаний (RAG) использует её как основной источник по вопросам ЖК.
- **AI в ежедневной сводке**
  - Формирует краткое резюме дня для админов без персональных данных.
- **AI-категоризация базы знаний**
  - При `/rag_bot` помогает автоматически проставить категорию и каноничное описание записи.

### Что контролирует админ

- `/ai_on` — включает runtime-флаг AI (если есть `AI_KEY` (или алиасы), бот работает через внешний провайдер).
- `/ai_off` — принудительно отключает runtime AI и переводит функции в локальный fallback.
- `/ai_status` — показывает текущий режим, usage за день и последнюю ошибку.
- `/ai_ping` — быстрый health-check клиента.
- `/ai_probe` — диагностика в 3 слоя: конфиг → реальный API-вызов → usage.
- `/ai_reset` — сбрасывает дневные счётчики AI-использования.

### Поведение при сбоях

- Если ключ не задан, API недоступно, превышены лимиты или временно отключён runtime-флаг — бот продолжает работать в локальном режиме.
- Критичные функции чата (модерация, команды, игры, статистика) при этом не останавливаются.

## Безопасность и секреты

**Правило: никаких паролей/токенов/ключей в открытом виде в репозитории.**

- Все секреты (`BOT_TOKEN`, `ANTHROPIC_API_KEY`, SSH-ключ, Docker Hub токен) хранятся
  **только** в GitHub Secrets. В коде, `docker-compose.yaml` и любых закоммиченных
  файлах их быть не должно — только ссылки вида `${{ secrets.* }}`.
- `.env` в git не коммитится (см. `.gitignore`). `.env.example` содержит только
  **пустые** поля и несекретные дефолты — как шаблон.
- На сервере секреты не лежат в `docker-compose.yaml`. Весь env-файл собирается из
  одного GitHub-секрета `BOT_ENV` и пишется деплоем в `/opt/alexbot/.env` (chmod 600).

## CI/CD

При пуше в `main` (или вручную через **Run workflow**) GitHub Actions собирает и
пушит Docker-образ, затем — если включён деплой — раскатывает его на сервер.

GitHub Secrets (Settings → Secrets and variables → Actions → **Secrets**):
- `DOCKERHUB_USERNAME` — логин Docker Hub
- `DOCKERHUB_TOKEN` — Access Token Docker Hub
- `SSH_HOST` / `SSH_USER` / `SSH_PRIVATE_KEY` — доступ к серверу для деплоя
- `BOT_ENV` — **всё содержимое `.env`** одним блоком (секреты + настройки).
  Источник истины для конфигурации сервера.

GitHub Variables (вкладка **Variables**):
- `DEPLOY_ENABLED=true` — включает джобу деплоя (пока сервер не готов — оставьте
  невыставленным, тогда собирается только образ).

Деплой запускается из Claude/вручную: Actions → *Build and Deploy* → **Run workflow**.

## Деплой на сервер (первичная настройка)

1. Поднять зарубежный VPS (рекомендация — Hetzner CX22, Германия/Финляндия),
   установить Docker + docker compose, создать каталог `/opt/alexbot`.
2. Добавить в GitHub секрет `BOT_ENV` (содержимое `.env.example` с заполненными
   значениями) и серверные секреты `SSH_*`, плюс переменную `DEPLOY_ENABLED=true`.
3. Запустить workflow — он соберёт образ, запишет `/opt/alexbot/.env` из `BOT_ENV`
   и поднимет контейнер. Данные бота хранятся в `/opt/alexbot/data` (SQLite).

Ручное обновление на сервере (если нужно):
```bash
cd /opt/alexbot
sh reload.sh          # docker compose pull && up -d
docker compose logs -f
```

## Локальный запуск

```bash
pip install -r requirements.txt
cp .env.example .env   # заполнить BOT_TOKEN, ANTHROPIC_API_KEY и т.д.
python -m app.main
```

## Как проверить на стенде (быстрый сценарий)

1. `cp .env.example .env` и заполнить обязательные переменные.
2. Запустить: `python -m app.main`.
3. В Telegram выполнить `/help` и проверить меню.
4. Выполнить `/ai_probe` и убедиться, что Anthropic API доступен.
5. Упомянуть бота в сообщении и проверить ответ ассистента.

## Почему бот может не запускаться

1. Не заполнен `.env` (минимум `BOT_TOKEN`, `FORUM_CHAT_ID`, `ADMIN_LOG_CHAT_ID`).
   На сервере прода бот в первую очередь подхватывает переменные из `/opt/alexbot/docker-compose.yaml` (сервис `bot`, секции `environment`/`env_file`) и использует их как источник по умолчанию, если переменные не выставлены в окружении процесса.
2. Бот был остановлен командой `/shutdown_bot` и остался файл-флаг `.stopped` в каталоге данных (`/app/data` или рядом с SQLite-файлом из `DATABASE_URL`).
3. Неверный `BOT_TOKEN` или нет доступа к Telegram API (на старте есть 3 попытки `get_me`).

Быстрая самопроверка:

```bash
cp .env.example .env
# заполните обязательные поля и перезапустите
python -m app.main
```
